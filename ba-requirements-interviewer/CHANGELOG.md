# Changelog
 
Все значимые изменения скилла `ba-requirements-interviewer` — ИИ-агента бизнес-аналитика для сбора требований в финтех-инвестициях.
 
Версии отражают итерации разработки в порядке появления. Даты не проставлены намеренно — при внедрении в git проставляются по фактическим коммитам.
 
---
 
## v0.8 — Видимость команд
 
- Памятка команд («пропустить · позже · статус · завершить сейчас») обязательна в первом сообщении сессии, повторяется в плане интервью и по команде «статус». Стейкхолдер не может пользоваться командами, о которых не знает.
- README.md и .gitignore для публикации скилла в git-репозитории.
## v0.7 — UX под опыт стейкхолдера/PO
 
Семь правок, снижающих трение для человека по ту сторону интервью.
 
- Тёплый старт: шаг 0.5 «Питч инициативы» — сначала «что хотите сделать и для кого», административные вопросы после.
- Код бюджета PPM (PQ-3) перенесён со старта в шаг 3, перед выдачей документа.
- Выбор темпа: режим «Интервью» (по вопросу) или «Бриф-дамп» (свободный рассказ, агент раскладывает по блокам).
- План и прогресс-индикатор: список блоков и ориентировочное время на старте, «блок N из M» в чекпоинтах (правило 16).
- Команды стейкхолдера: `пропустить`, `позже`, `статус`, `завершить сейчас`.
- Право на «не знаю»: нормализация для блоков 6–8; делегируемые списки вопросов по ролям (комплаенс, архитектор, ИБ, операции) как приложение к BRD.
- Приоритет решений стейкхолдера (правило 15): агент не блокирует запись требования; методологические правила стали рекомендациями с тегом `[зафиксировано под ответственность стейкхолдера]`.
- Схлопнутый финал: каскад проверок заменён «Консолидированным ревью» (одна таблица), фидбек включён в финальное сообщение.
## v0.6 — Механика работы агента
 
- Возобновление прерванного интервью: третий режим в шаге 0 + промежуточное сохранение черновика после каждого блока.
- Pre-read артефактов (шаг 1.5): разбор существующих материалов и предзаполнение ответов тегом `[гипотеза из документа]`; защита от инъекций из загруженных документов.
- Скоринг готовности (Definition of Ready): метрики зрелости и вердикт «готов к передаче в SA» / «требует дозаполнения».
- Петля обратной связи (шаг 4): три вопроса стейкхолдеру + наблюдения агента, сохраняются рядом с BRD как сырьё для ревизии банка вопросов.
## v0.5 — Форматирование банка вопросов
 
- Сквозная нумерация вопросов в формате `{блок}.{вопрос}` (1.1, 1.2, …).
- Каждый вопрос вынесен на отдельную строку для корректного рендеринга.
## v0.4 — Каналы выдачи результата
 
Выбор из трёх каналов вместо жёсткого сохранения в git:
 
- Git — запрос ссылки на ветку, формат markdown.
- Confluence — запрос родительской страницы, формат markdown.
- Файл на скачивание — уточнение формата разметки (md / txt / html / другое).
## v0.3 — Усиление по лучшим практикам БА
 
- Карта стейкхолдеров и тег `[требует кросс-валидации]` для единственного источника.
- MoSCoW-приоритизация отдельным проходом в конце интервью.
- Трассируемость требований к целям/KPI; проверки «сироты» и «пустые цели».
- Критерии качества формулировки (атомарность, проверяемость, запрет неквантифицированных наречий).
- Walkthrough итогового документа и статусная модель «черновик → на согласовании → утверждён» с подписантами.
- Секция «Глоссарий и допущения» в шаблоне.
## v0.2 — Проектный контекст, git, режимы
 
- Стартовые проектные вопросы (`project-context-questions.md`): юрлицо (Альфа-Инвестиции / ГоИнвест / оба), корпоративные KPI, код бюджета PPM и подраздел КР — в формате опций со свободным вводом.
- Сохранение результирующего документа в git.
- Выбор режима в начале сессии: новое требование / продолжение существующего.
## v0.1 — Первичная упаковка скилла
 
- Роль агента: бизнес-аналитик с опытом в российском брокеридже.
- Правила ведения интервью: один вопрос за раз, воронка по блокам, отделение требования от решения, тегирование достоверности, чекпоинты, open questions с владельцами.
- `references/question-bank.md`: 65 вопросов в 10 блоках + таблица follow-up паттернов.
- `references/output-template.md`: шаблон итогового документа требований (BRD).
